# SDD 常见问题与解答

---

## 1. 有没有 skill 通过大模型问答自动生成 spec？

**目前没有现成的，但可以立刻创建一个 `/spec-new` skill。** 设计思路：

```
用户输入: /spec-new
→ Skill 按照template模式提问：
   1. 属于哪个 domain？(auth / system / platform / 新建?)
   2. 用一句话描述这个模块要做什么？
   3. 大致涉及哪些数据字段和接口？
→ AI 自动：
   - 分配下一个 NNN 编号
   - 复制模板 → 根据问答填充 §1~§10 的初稿
   - 未决问题标出来让用户确认
   - 跑 specs:validate 确保格式正确
→ 输出: 一份可评审的 draft spec
```

**关键点**：AI 生成的是 **draft**，必须人工评审后才能 `approved`。AI 负责结构化和一致性，人负责正确性。

## 2. 如何保持文档与代码的强一致？

三道防线：

| 层级 | 机制 | 现在有没有 |
|------|------|-----------|
| **提交时** | commit message 必须带 `Refs: SYS-xxx`；PreCommit hook 跑 `specs:validate` | 已配置校验脚本，hook 可一键接入 |
| **CR 时** | PR 描述里对照 spec §4/§6/§7 逐条勾选（PR 模板见 `USAGE.md §5.5`） | 模板已写好，放 `.github/` 即可 |
| **运行时** | 接口契约测试：从 spec §7 自动生成请求/断言骨架，CI 跑真实接口 | 尚未做（需后端跑起来后落地） |

**最狠的一招**（推荐后期加）：
- 写一个脚本/skill 扫描 `src/api/*.js` + `src/mock/api/*.js` 的路由，与 spec §7 的表格做 diff。路由/字段不一致直接报错。这就把"文档 ↔ 代码"的漂移变成了编译器级别的硬错误。

## 3. SDD + AI 如何优化 spec 结构？

当前 spec 的 10 个章节是通用模板。AI 可以在两个方向优化：

**A. 渐进式写作（推荐）**

不是一口气写完 10 章，而是分阶段：
```
Phase 1: 只写 §1 背景 + §3 用户故事 + §10 未决问题（5 分钟）
Phase 2: AI 根据 Phase 1 自动草拟 §4 功能需求 + §9 验收标准
Phase 3: 人确认后，AI 自动推导 §6 数据契约 + §7 接口契约
Phase 4: 人补充 §2 范围 + §5 非功能 + §8 权限边界
```
→ **用户只需要写"我要什么"（故事），AI 负责推导"契约长什么样"**。

**B. Spec 之间自动关联**

AI 在你写新 spec 时自动：
- 检查 `depends-on` 应该引用谁（扫描字段/接口重叠）
- 检查是否与已有 spec 产生冲突（如字段名碰撞）
- 建议哪些旧 spec 需要 bump version

这些都可以做成 `/spec-review` skill。

## 5. 文件是不是太多？每次小改都多一个文件？

**不会。** 规则是：

| 变更类型 | 要做什么 | 新增文件数 |
|----------|----------|-----------|
| 小改（加字段、改措辞） | 只改已有 spec 的内容 + bump patch | **0** |
| 中改（加接口、改逻辑） | 只改已有 spec + bump minor | **0** |
| 大改（破坏性变更） | 改 spec + bump major + 写 ADR | **+1**（ADR） |
| 新模块 | 新建 spec + plan + tasks | **+3** |
| 推翻旧决策 | 改旧 ADR + 写新 ADR | **+1** |

**90% 的日常修改是小改和中改——0 新增文件**，只是在已有 `*.spec.md` 里改内容 + bump version + 追加一行 Changelog。

plan 和 tasks 也不是每个 spec 都必须有——只有要开始**编码**时才写。纯草稿阶段的 spec 只有一个 `.spec.md` 文件。

**文件总量对照**：
```
当前 specs/ 目录：17 个文件（含模板、脚本、README）
对比旧方式：4 个大杂烩文档（需求文档 + plan + 后端计划 + ai/）
```

文件多了 13 个，但换来的是：**每个文件独立可审、可追溯、可校验、可并行开发**。大杂烩文档越长越难维护；独立小文件反而更易治理。

## 附录：bump patch / minor / major 是什么？

这是 **语义化版本（Semantic Versioning）** 的术语，版本号格式为：

```
MAJOR.MINOR.PATCH
  ↑      ↑     ↑
 大版本  小版本  补丁
```

"bump" = "递增一位"。

| 操作 | 意思 | 版本变化示例 | 什么时候用 |
|------|------|-------------|-----------|
| **bump patch** | 补丁 +1 | `0.1.0` → `0.1.1` | 改错别字、补充说明、修措辞（不影响行为） |
| **bump minor** | 小版本 +1，patch 归零 | `0.1.1` → `0.2.0` | 加字段、加接口、加功能（**向后兼容**，旧的不受影响） |
| **bump major** | 大版本 +1，其余归零 | `0.2.0` → `1.0.0` | 删字段、改语义、改表结构（**破坏性变更**，旧代码必须跟着改） |

**对应到 spec 里**：

```markdown
## Changelog
| 0.1.0 | 初稿 |                       ← 首次创建
| 0.1.1 | 修正"用户名"长度描述 |      ← bump patch（纯文字修正）
| 0.2.0 | 新增 avatar 字段 |          ← bump minor（加东西，旧的不破坏）
| 1.0.0 | 字典改为树形结构 |          ← bump major（破坏性，旧逻辑全要改）
```

简单记：**patch 修文字，minor 加东西，major 破坏旧的**。
