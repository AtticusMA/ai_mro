# 智慧机务系统 - 完整需求文档

## 1. 项目概述

### 1.1 项目名称
智慧机务系统（AI 工作辅助系统）

### 1.2 开发策略
**前端先行，后端跟进**
- 第一阶段：前端完整开发（包含 Mock 数据和动态演示）
- 第二阶段：后端开发（与前端集成）
- 优势：快速看到效果、及时调整需求、前后端并行开发

### 1.3 技术栈
- **前端**：Vue.js 3.x + Tailwind CSS + Element Plus
- **后端**：Java Spring Boot + Redis + Nacos + Dubbo 微服务框架
- **数据库**：MySQL 8.0
- **认证方式**：用户名/密码登录 + JWT Token + RBAC权限模型
- **前端 Mock**：Mock.js（开发阶段模拟后端数据）

### 1.4 核心特性
- 完整的权限管理系统（RBAC）
- 灵活的数据权限过滤机制（5种权限类型）
- 树形部门结构管理
- 用户、角色、菜单权限的完整管理
- 字典数据管理
- 安全的认证和授权
- 前端 Mock 数据支持（开发阶段）
- 动态演示页面（展示系统功能）

### 1.5 首期范围外（Out of Scope）

- AI 业务能力（后续按 Spec 增量加入）
- SSO 登录（仅用户名/密码）
- 多租户、国际化（保留扩展点）
- 替代既有 OA / HR 系统
- C 端用户能力

### 1.6 关键里程碑

| 阶段 | 目标 |
|------|------|
| M1 | 前端骨架 + Mock + 6 大模块 UI 跑通 |
| M2 | 后端认证 + 用户/部门 + JWT |
| M3 | 角色/菜单/字典 + 数据权限拦截器 |
| M4 | 前后端联调 + 安全/性能验收 |

## 2. 系统架构

### 2.1 微服务模块划分
```
gateway-service          # API网关
├── auth-service        # 认证服务（登录、Token管理）
├── system-service      # 系统管理服务（用户、部门、角色、菜单、字典）
└── business-service    # 业务服务（后续扩展）
```

### 2.2 安全设计
- **密码加密**：BCrypt（替代不安全的MD5）
- **认证方式**：JWT Token
- **授权模型**：RBAC（基于角色的访问控制）
- **数据权限**：动态SQL过滤（通过MyBatis拦截器实现）
- **防护措施**：XSS防护、SQL注入防护、接口防重放

## 3. 管理后台功能模块

### 3.1 登录模块

#### 3.1.1 登录页面
- 用户名输入框
- 密码输入框
- 记住密码选项
- 登录按钮
- 验证码（可选）

#### 3.1.2 登录流程
1. 用户输入用户名和密码
2. 后端验证用户名和密码（使用BCrypt比对）
3. 验证成功后生成JWT Token
4. 返回Token和用户基本信息
5. 前端存储Token到localStorage/sessionStorage
6. 后续请求在Authorization Header中携带Token

#### 3.1.3 登出
- 清除前端Token
- 后端可选：将Token加入黑名单（Redis）

## 4. 数据权限机制详解

### 4.1 五种数据权限类型

#### 4.1.1 全部数据（超管）
- 权限类型值：1
- 说明：超级管理员可查看系统中所有数据
- 过滤条件：无（不添加任何WHERE条件）

#### 4.1.2 本部门数据
- 权限类型值：2
- 说明：只能查看本部门创建的数据
- 过滤条件：`WHERE create_dept_id = #{userDeptId} OR use_for_dept_id = #{userDeptId}`

#### 4.1.3 本部门及子部门数据
- 权限类型值：3
- 说明：可查看本部门及所有子部门创建的数据
- 过滤条件：`WHERE create_dept_id IN (#{deptIdList}) OR use_for_dept_id IN (#{deptIdList})`
  - 其中deptIdList包含：用户所在部门 + 该部门的所有子部门

#### 4.1.4 本人数据
- 权限类型值：4
- 说明：只能查看自己创建的数据
- 过滤条件：`WHERE create_user_id = #{userId}`

#### 4.1.5 自定义部门数据
- 权限类型值：5
- 说明：可查看指定部门及其子部门创建的数据
- 过滤条件：`WHERE create_dept_id IN (#{customDeptIdList}) OR use_for_dept_id IN (#{customDeptIdList})`
  - 其中customDeptIdList由管理员在角色配置中指定

### 4.2 数据权限配置流程

#### 4.2.1 为用户分配角色时
1. 选择用户
2. 选择要分配的角色
3. 系统自动获取该角色的数据权限类型
4. 如果角色的数据权限类型=5（自定义部门），则显示部门树供选择
5. 用户可多选部门，选中部门时其子部门自动选中
6. 保存用户-角色关联关系

#### 4.2.2 用户拥有多个角色时
- 系统取最宽松的权限类型
- 权限宽松度排序：全部数据 > 本部门及子部门 > 本部门 > 自定义部门 > 本人
- 例如：用户同时拥有"本部门"和"本部门及子部门"角色，则取"本部门及子部门"

### 4.3 数据权限过滤实现

#### 4.3.1 业务数据表结构要求
所有需要进行数据权限过滤的业务表，必须包含以下字段：
- `create_user_id` - 创建人ID
- `create_dept_id` - 创建人所在部门ID
- `use_for_dept_id` - 指定其他部门可见（可为NULL）
- `create_time` - 创建时间
- `update_user_id` - 更新人ID
- `update_time` - 更新时间

#### 4.3.2 MyBatis拦截器实现
1. 拦截所有SELECT语句
2. 获取当前登录用户信息（从ThreadLocal或SecurityContext）
3. 查询用户的所有角色及数据权限类型
4. 根据权限类型构建WHERE条件
5. 动态修改SQL，添加过滤条件
6. 执行修改后的SQL

#### 4.3.3 部门树查询算法
- 使用递归CTE（Common Table Expression）快速查询部门树
- 或使用ancestors字段（存储祖级部门ID列表）进行快速查询
- 缓存部门树结构到Redis，减少数据库查询

#### 4.3.4 SQL过滤示例

**场景1：用户权限类型=2（本部门）**
```sql
-- 原始SQL
SELECT * FROM business_table WHERE status = 1;

-- 过滤后的SQL
SELECT * FROM business_table 
WHERE status = 1 
AND (create_dept_id = 10 OR use_for_dept_id = 10);
```

**场景2：用户权限类型=3（本部门及子部门）**
```sql
-- 原始SQL
SELECT * FROM business_table WHERE status = 1;

-- 过滤后的SQL
SELECT * FROM business_table 
WHERE status = 1 
AND (create_dept_id IN (10, 11, 12, 13) OR use_for_dept_id IN (10, 11, 12, 13));
-- 其中10是用户部门，11,12,13是其子部门
```

**场景3：用户权限类型=5（自定义部门）**
```sql
-- 原始SQL
SELECT * FROM business_table WHERE status = 1;

-- 过滤后的SQL
SELECT * FROM business_table 
WHERE status = 1 
AND (create_dept_id IN (10, 20, 30, 31, 32) OR use_for_dept_id IN (10, 20, 30, 31, 32));
-- 其中10,20,30是管理员指定的部门，31,32是30的子部门
```

## 10. 服务端日志功能

### 10.1 概述

服务端日志分两个维度：
1. **应用运行日志**：由 Log4j2 框架按级别分文件滚动输出，用于运维排障与性能分析。
2. **业务操作审计日志**：由 Spring AOP 拦截器采集，持久化到数据库，支持前端查询（详见 SYS-007 Spec）。

适用服务：`manage-web`、`auth-service`、`system-service` 及所有 MRO 微服务。

---

### 10.2 应用运行日志（Log4j2）

#### 10.2.1 框架选型

- 日志框架：**Log4j2**（`log4j2.xml` 配置文件）
- SLF4J 作为门面，底层绑定 `log4j-slf4j2-impl`
- 配置热重载：`monitorInterval="30"`（秒），运行时修改配置无需重启

#### 10.2.2 日志级别策略

| 级别 | 使用场景 |
|------|---------|
| ERROR | 系统异常、未捕获错误、第三方接口调用失败 |
| WARN | 业务告警、降级处理、接口耗时超阈值 |
| INFO | 请求入口/出口、关键业务节点、启动/关闭事件 |
| DEBUG | SQL 语句（Mapper 层）、详细参数、流程分支 |

根日志级别设为 **INFO**；Mapper 层（MyBatis 生成的 SQL）使用异步 **TRACE** 级别单独采集。

#### 10.2.3 日志文件输出规范

所有服务的日志统一输出到 `/opt/logs/{服务名}/` 目录，按级别分四个滚动文件：

| 文件名 | 过滤级别 | 最大单文件大小 | 最大保留数 | 归档目录 |
|--------|---------|--------------|-----------|---------|
| `debug.log` | DEBUG 及以上 | 200 MB | 30 个 | `$${date:yyyy-MM}/` |
| `info.log` | INFO 及以上 | 200 MB | 默认 7 | `$${date:yyyy-MM}/` |
| `warn.log` | WARN 及以上 | 200 MB | 20 个 | `$${date:yyyy-MM}/` |
| `error.log` | ERROR 及以上 | 200 MB | 默认 7 | `$${date:yyyy-MM}/` |

归档命名格式：`{服务名}-{级别}-%d{yyyy-MM-dd}-%i.log`，按天滚动（`TimeBasedTriggeringPolicy interval="1"`）。

#### 10.2.4 日志输出格式

```
%d{yyyy-MM-dd HH:mm:ss:SSS} | %p | - %l - %m%n
```

| 占位符 | 含义 |
|--------|------|
| `%d{...}` | 时间戳，精确到毫秒 |
| `%p` | 日志级别 |
| `%l` | 输出位置（类名.方法名:行号） |
| `%m` | 日志消息 |
| `%n` | 换行 |

#### 10.2.5 异步 Logger（Mapper 层）

MyBatis Mapper 接口产生的 SQL 日志量大，使用 `AsyncLogger` 单独处理，避免阻塞主线程：

```xml
<AsyncLogger name="com.mro.{service}.mapper" level="TRACE" additivity="false">
    <appender-ref ref="Console"/>
    <appender-ref ref="RollingFileDebug"/>
</AsyncLogger>
```

- `additivity="false"`：不向 root logger 传播，避免重复输出
- 生产环境可将 `level` 调整为 `INFO` 关闭 SQL 日志

#### 10.2.6 log4j2.xml 参考配置结构

```xml
<configuration status="WARN" monitorInterval="30">
  <appenders>
    <!-- 控制台输出 -->
    <Console name="Console" target="SYSTEM_OUT">
      <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss:SSS} | %p | - %l - %m%n"/>
    </Console>

    <!-- DEBUG 分级滚动文件 -->
    <RollingFile name="RollingFileDebug"
                 fileName="/opt/logs/{服务名}/debug.log"
                 filePattern="/opt/logs/{服务名}/$${date:yyyy-MM}/{服务名}-debug-%d{yyyy-MM-dd}-%i.log">
      <ThresholdFilter level="debug" onMatch="ACCEPT" onMismatch="DENY"/>
      <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss:SSS} | %p | - %l - %m%n"/>
      <Policies>
        <TimeBasedTriggeringPolicy interval="1" modulate="true"/>
        <SizeBasedTriggeringPolicy size="200MB"/>
      </Policies>
      <DefaultRolloverStrategy max="30"/>
    </RollingFile>

    <!-- INFO / WARN / ERROR 结构相同，ThresholdFilter level 不同 -->
    <!-- RollingFileInfo: level="info", RollingFileWarn: level="warn", RollingFileError: level="error" -->
  </appenders>

  <loggers>
    <!-- 屏蔽 Spring/MyBatis 框架 DEBUG 噪声 -->
    <Logger name="org.springframework" level="INFO"/>

    <!-- 根 Logger -->
    <Root level="INFO">
      <AppenderRef ref="Console"/>
      <AppenderRef ref="RollingFileInfo"/>
      <AppenderRef ref="RollingFileDebug"/>
      <AppenderRef ref="RollingFileWarn"/>
      <AppenderRef ref="RollingFileError"/>
    </Root>

    <!-- Mapper 层异步 SQL 日志 -->
    <AsyncLogger name="com.mro.{service}.mapper" level="TRACE" additivity="false">
      <AppenderRef ref="Console"/>
      <AppenderRef ref="RollingFileDebug"/>
    </AsyncLogger>
  </loggers>
</configuration>
```

### 10.3 业务操作审计日志（SYS-007）

> 详细 Spec 见 `specs/system/007-operation-log.spec.md`（SYS-007 v0.2.0）。

审计日志同时满足两个目标：
1. **控制台实时打印**：开发/运维可在日志文件中通过 requestId 串联一次完整请求的全链路日志。
2. **数据库异步持久化**：支持前端查询与数据权限过滤。

---

#### 10.3.1 requestId 生成与传递

- requestId 由**前端生成**，通过 HTTP 请求 Header `X-Request-Id` 传入服务端，服务端不负责生成。
- `manage-web` 入口 Filter（`CachedBodyFilter` 或专用 `RequestIdFilter`）从 Header 中读取 requestId，写入 **MDC**（`MDC.put("requestId", id)`），Log4j2 PatternLayout 通过 `%X{requestId}` 自动输出到每行日志。
- 若请求未携带 `X-Request-Id` Header，则 MDC 中 requestId 记为 `-`，不影响正常流程。
- manage-web 调用 Dubbo 时，将 requestId 注入 **RpcContext Attachment**（key: `requestId`），下游微服务从 Attachment 读取后同样写入本地 MDC。
- 请求结束时执行 `MDC.clear()`，避免线程池复用导致 MDC 污染。

Log4j2 格式需在 PatternLayout 中加入 requestId：

```
%d{yyyy-MM-dd HH:mm:ss:SSS} | %p | [%X{requestId}] | - %l - %m%n
```

---

#### 10.3.2 HTTP 拦截器控制台打印

在 `manage-web` 的 `interceptor/OperationLogInterceptor` 中，分**执行前**和**执行后**两阶段打印：

**执行前（`preHandle`）— INFO 级别**

```
[REQ-IN ] requestId=<uuid> | path=<请求路径> | time=<yyyy-MM-dd HH:mm:ss:SSS> | userId=<操作人ID> | body=<请求体JSON，超512字符截断>
```

示例：
```
2026-05-28 10:23:45:123 | INFO | [a1b2c3d4] | - OperationLogInterceptor:45 -
[REQ-IN ] requestId=a1b2c3d4 | path=/api/user/list | time=2026-05-28 10:23:45:123 | userId=100 | body={"pageNum":1,"pageSize":20}
```

**执行后（`afterCompletion`）— INFO 级别**

```
[REQ-OUT] requestId=<uuid> | result=<业务响应JSON，超1024字符截断> | costTime=<毫秒>ms | status=<success|fail>
```

示例：
```
2026-05-28 10:23:45:456 | INFO | [a1b2c3d4] | - OperationLogInterceptor:78 -
[REQ-OUT] requestId=a1b2c3d4 | result={"code":0,"msg":"ok","data":{...}} | costTime=333ms | status=success
```

字段说明：

| 字段 | 来源 | 说明 |
|------|------|------|
| requestId | MDC | 本次请求唯一标识，用于跨日志行关联 |
| path | `HttpServletRequest.getRequestURI()` | 请求路径 |
| time | 拦截器记录的请求开始时间 | 格式 `yyyy-MM-dd HH:mm:ss:SSS` |
| userId | `UserContext.getUserId()` | 来自 Gateway 注入的 Header，匿名请求记 `-` |
| body | 请求体 + query params 合并 JSON | 超 512 字符截断并追加 `...[truncated]` |
| result | 响应体 JSON（通过 `ContentCachingResponseWrapper` 读取） | 超 1024 字符截断 |
| costTime | `afterCompletion` 时间 - `preHandle` 时间 | 毫秒 |
| status | HTTP 状态码 < 400 且业务 code=0 为 `success`，否则 `fail` | |

---

#### 10.3.3 Dubbo 过滤器控制台打印

在每个微服务（`system-service`、`auth-service` 等）中实现 **Dubbo Filter**（`com.mro.{service}.filter.DubboLogFilter`），在 Provider 端打印：

**调用前（`invoke` 方法执行前）— INFO 级别**

```
[DUBBO-IN ] requestId=<uuid> | method=<接口类名>#<方法名> | params=<参数JSON，超512字符截断>
```

示例：
```
2026-05-28 10:23:45:234 | INFO | [a1b2c3d4] | - DubboLogFilter:38 -
[DUBBO-IN ] requestId=a1b2c3d4 | method=UserService#getUserById | params=[100]
```

**调用后（正常返回）— INFO 级别**

```
[DUBBO-OUT] requestId=<uuid> | method=<接口类名>#<方法名> | costTime=<毫秒>ms | result=<返回值JSON，超512字符截断>
```

**调用后（异常）— ERROR 级别**

```
[DUBBO-ERR] requestId=<uuid> | method=<接口类名>#<方法名> | costTime=<毫秒>ms | error=<异常类型: 异常消息>
```

字段说明：

| 字段 | 来源 | 说明 |
|------|------|------|
| requestId | `RpcContext.getServerAttachment("requestId")` | manage-web 透传，不存在时记 `-` |
| method | `invoker.getInterface().getSimpleName() + "#" + invocation.getMethodName()` | Dubbo RPC 方法标识 |
| params | `invocation.getArguments()` 序列化为 JSON | 超 512 字符截断 |
| costTime | Filter 内计时 | 毫秒 |
| result | 返回值序列化为 JSON | 超 512 字符截断 |

Dubbo Filter 注册方式（SPI）：在 `resources/META-INF/dubbo/com.alibaba.dubbo.rpc.Filter` 中声明：
```
dubboLogFilter=com.mro.{service}.filter.DubboLogFilter
```
并在 `@DubboService` 全局配置 `filter="dubboLogFilter"`。

#### 10.3.4 数据库异步持久化

`afterCompletion` 阶段，HTTP 拦截器额外将本次请求信息**异步**写入 `sys_operation_log` 表（线程池执行，写入失败静默捕获）：

| 字段 | 类型 | 说明 |
|------|------|------|
| request_path | VARCHAR(512) | 请求路径 |
| request_time | DATETIME(3) | 请求发生时间（毫秒精度） |
| cost_time | BIGINT | 请求耗时（毫秒） |
| user_id | BIGINT | 操作人用户 ID（来自 UserContext） |
| user_name | VARCHAR(64) | 操作人姓名 |
| request_params | TEXT | 请求参数 JSON（超 4096 字符截断） |
| result_status | TINYINT | 操作结果：1=成功，0=失败 |
| dept_id | BIGINT | 操作人所属部门 ID（用于 SYS-006 数据权限过滤） |

> 控制台打印与数据库写入共用同一次拦截，参数只解析一次。

---

#### 10.3.5 查询接口

| Method | Path | 权限标识 | 说明 |
|--------|------|---------|------|
| GET | `/api/system/operation-log` | `log:list` | 分页查询（支持操作人/时间范围/路径/结果过滤） |
| GET | `/api/system/operation-log/{id}` | `log:list` | 查询单条详情（含完整请求参数和耗时） |

---

#### 10.3.6 非功能要求

- 控制台打印与数据库写入共用拦截器，请求参数只解析一次，不重复消耗性能。
- 异步写入不使主业务接口响应时间增加超过 10ms；控制台打印为同步但开销可忽略（仅字符串拼接）。
- `request_time` 字段建立索引，列表查询 P95 < 1s。
- 日志只读，**不提供删除/修改接口**。
- 日志访问须携带有效 JWT，数据权限遵循 SYS-006 五种类型过滤。
- Dubbo Filter 异常情况不影响业务正常返回（try-finally 保护）。
